## 二、遗漏功能清单（按优先级）

### 🔴 P0 — 核心缺失（强烈建议尽快补全）

#### 1. 标签系统（Tag System）
- **为什么重要：** Flomo 的灵魂功能。万物手札目前只有"分类"（单选），但实际场景中一个物品/记录可以打多个标签（如 `#厨房 #调料 #快过期`）
- **竞品参考：** Flomo 的 `#标签` 语法、Notion 的 multi-select
- **实现建议：**
  - 记录编辑页增加"标签"输入框，支持 `#` 触发自动补全
  - 标签数据存为字符串数组 `tags: []`
  - 搜索页支持按标签筛选（点击标签直接过滤）
  - 标签云展示在统计页
- **工作量：** 小（~200 行代码）

#### 2. 过期/保质期提醒
- **为什么重要：** 物品管理 App 的刚需。Sorter、过期啦、Grocy 都有
- **竞品参考：** 过期啦的扫码 + 保质期、Grocy 的到期日追踪
- **实现建议：**
  - 记录表单增加"保质期"字段（天数或具体日期）
  - 首页增加"即将过期"筛选标签（3 天内/7 天内）
  - 使用 `Notification API` 推送浏览器通知（需用户授权）
  - 列表项显示过期状态（🟢正常 即将过期 🔴已过期）
- **工作量：** 中（~300 行代码）

#### 3. 数量管理 & 库存追踪
- **为什么重要：** 物品管理区别于笔记的核心。用户需要知道"家里还有几包纸巾"
- **竞品参考：** Sorter 的数量加减、Grocy 的消耗/采购记录
- **实现建议：**
  - 记录增加 `quantity` 字段（默认 1）
  - 详情页增加 `+` / `-` 快捷按钮，快速加减数量
  - 记录每次数量变更的历史（`quantityHistory: [{date, delta, reason}]`）
- **工作量：** 小（~150 行代码）

---

### 🟠 P1 — 重要增强（提升用户体验）

#### 4. 多视图切换（列表/卡片/时间线）
- **为什么重要：** Notion 的杀手级功能。同一份数据，不同场景需要不同展示
- **竞品参考：** Notion 的 List / Board / Gallery / Timeline 视图
- **实现建议：**
  - 顶部增加视图切换按钮（列表/卡片/时间线）
  - 列表视图：当前样式
  - 卡片视图：照片 + 标题 + 标签的卡片网格
  - 时间线视图：按日期分组的纵向时间轴
- **工作量：** 中（~400 行代码）

#### 5. 模板系统
- **为什么重要：** 减少重复录入。用户经常记录同类物品（如日用品补货）
- **竞品参考：** Notion 模板、纯纯写作的写作模板
- **实现建议：**
  - 记录详情页增加"保存为模板"按钮
  - 创建页增加"从模板创建"入口
  - 模板存为独立集合 `templates: [{name, category, tags, fields}]`
  - 预设几个常用模板（日用品、食品、电子产品等）
- **工作量：** 小（~200 行代码）

#### 6. 版本历史（Undo / 操作日志）
- **为什么重要：** 纯纯写作的亮点。用户误删/误改后可以恢复
- **竞品参考：** 纯纯写作的版本历史、Notion 的页面历史
- **实现建议：**
  - 每次编辑时自动保存旧版本快照（保留最近 10 个版本）
  - 详情页增加"历史记录"入口
  - 支持一键恢复到任意历史版本
  - 存储格式：`history: [{timestamp, data}]`
- **工作量：** 小（~200 行代码）

#### 7. 地理位置 & 天气标注
- **为什么重要：** Day One 的自动标注让记录更有"场景感"
- **竞品参考：** Day One 的 GPS + 天气自动标注
- **实现建议：**
  - 创建/编辑记录时可选"添加位置"（调用 `navigator.geolocation`）
  - 显示位置名称（可用逆地理编码 API，如高德/腾讯免费 API）
  - 可选添加当天天气（调用免费天气 API）
  - 位置信息存储在记录的 `location` 字段
- **工作量：** 中（~300 行代码，需接入 API）

### 🟡 P2 — 体验优化（锦上添花）

#### 8. 思维导图 / 关系图谱
- **为什么重要：** Flomo 的思维导图让碎片笔记产生关联
- **竞品参考：** Flomo 的思维导图视图
- **实现建议：**
  - 使用 `ECharts` 的关系图（已有依赖）或 `D3.js`
  - 节点 = 记录，边 = 共同标签/同分类
  - 统计页增加"关系图谱"Tab
- **工作量：** 中（~300 行代码，ECharts 已有）

#### 9. 扫码录入（条形码/二维码）
- **为什么重要：** Grocy、过期啦的核心功能。扫一下就能录入商品信息
- **竞品参考：** Grocy 的 Barcode lookup、过期啦的扫码
- **实现建议：**
  - 使用 `html5-qrcode` 或 `QuaggaJS` 库
  - 创建页增加"扫码"按钮
  - 扫码后自动填充物品名称（可对接公开商品数据库，如 Open Food Facts）
- **工作量：** 中（~250 行代码 + 引入库）

#### 10. 分享功能（生成图片卡片）
- **为什么重要：** 用户想分享物品/收藏给好友
- **竞品参考：** 小红书卡片分享、Flomo 的分享链接
- **实现建议：**
  - 使用 `html2canvas` 将记录渲染为图片
  - 支持选择分享模板（简约/文艺/卡片）
  - 生成图片后调用 `navigator.share()` 分享
- **工作量：** 小（~150 行代码 + html2canvas）

#### 11. 快捷键支持（桌面端）
- **为什么重要：** 提升桌面端操作效率
- **竞品参考：** Notion 的全键盘操作
- **实现建议：**
  - `Ctrl+N` 新建、`Ctrl+F` 搜索、`Ctrl+S` 保存
  - `↑/↓` 切换记录、`Enter` 查看详情
  - 设置页增加"快捷键帮助"弹窗（`?` 触发）
- **工作量：** 小（~100 行代码）

### 🟢 P3 — 可选增强（长期规划）

#### 12. PWA 完整支持（离线使用）
- **为什么重要：** 当前已有 meta 标签，但缺 Service Worker
- **实现建议：**
  - 添加 `manifest.json`
  - 注册 Service Worker 缓存核心资源
  - 支持离线浏览和基础 CRUD
  - 在线时同步数据
- **工作量：** 中（~400 行代码）

#### 13. 云同步 / 多设备
- **为什么重要：** 当前数据仅存 localStorage，换设备就没了
- **实现建议：**
  - 方案 A（轻量）：对接 WebDAV（如坚果云）做定时同步
  - 方案 B（完整）：搭建后端 + 数据库（Supabase / LeanCloud）
  - 方案 C（最简单）：加密后同步到 GitHub Gist
- **工作量：** 大（需后端架构）

#### 14. AI 智能分类 & 标签推荐
- **为什么重要：** 降低用户录入成本
- **实现建议：**
  - 调用免费 LLM API（如通义千问）
  - 输入物品名称后自动推荐分类和标签
  - 本地维护一个关键词→标签的映射表（轻量方案）
- **工作量：** 中（~200 行代码 + API 调用）

#### 15. 骨架屏 & 加载动画
- **为什么重要：** 提升首屏感知性能
- **实现建议：**
  - 记录列表使用 CSS 骨架屏（灰色占位块）
  - 图片懒加载 + 占位图
  - 数据加载时显示骨架屏，加载完成后淡入
- **工作量：** 小（~100 行 CSS）

## 四、技术债务提醒

| 问题 | 风险 | 建议 |
|------|------|------|
| Base64 照片存储 | localStorage 5MB 限制，几张高清照片就满了 | 改用 IndexedDB（50MB+）或压缩图片 |
| 无数据校验 | 导入 JSON 可能注入非法数据 | 增加 schema 校验（Zod 或手动） |
| 密码加密弱 | CryptoJS AES 在客户端，密钥硬编码 | 改用 PBKDF2 派生密钥 + 用户密码 |
| 无错误边界 | 一个 JS 错误可能白屏 | 添加 try-catch + 错误提示 UI |

## 五、核心洞察

**当前最大的三个缺口：**

1. 🏷️ **标签系统** — 手札类产品的灵魂，没有标签就等于没有知识关联
2. ⏰ **过期提醒** — 物品管理区别于笔记的核心场景
3. 🔢 **数量管理** — 物品管理的刚需，"家里还有几包"是高频问题

**这三个建议优先在 Phase 1 完成，完成后产品定位会从"笔记工具"真正转变为"物品管理工具"。**
