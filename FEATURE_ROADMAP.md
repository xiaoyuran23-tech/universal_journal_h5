## 二、功能规划矩阵

### P0 - 核心体验提升（优先级最高）

#### 1. 数据持久化：IndexedDB 完整实现
- **背景**：当前仅使用 `localStorage`，存储上限约 5-10MB，且同步操作阻塞主线程
- **方案**：
  - 使用 `idb` 库（轻量级 IndexedDB Promise 封装）
  - 实现懒加载：首页仅加载最近 50 条，滚动时按需加载
  - 图片压缩存储：Canvas 压缩至 800px 宽，WebP 格式
- **可行性**：⭐⭐⭐⭐⭐ 高（IndexedDB 为 Web 标准，兼容性极好）
- **工作量**：2-3 天

#### 2. 标签系统（Tags）
- **背景**：当前仅有"分类"（单级），同类应用普遍支持多标签
- **方案**：
  - 每条记录支持多个标签（如 `#旅行 #美食 #2024`）
  - 标签云展示（按使用频率排序）
  - 标签筛选（点击标签过滤列表）
- **可行性**：⭐⭐⭐⭐⭐ 高（纯数据结构变更）
- **工作量**：1-2 天

#### 3. 图片压缩与懒加载
- **背景**：当前照片直接存储 Base64，极易撑爆 localStorage
- **方案**：
  - 上传时自动压缩（Canvas API）
  - 列表页使用缩略图，详情页加载原图
  - 超出存储配额时自动清理旧图片缓存
- **可行性**：⭐⭐⭐⭐ 高
- **工作量**：2 天

---

### P1 - 体验增强（重要但不紧急）

#### 4. 日历视图
- **背景**：手帐类应用标配功能，按日期浏览记录
- **方案**：
  - 集成轻量级日历组件（如 `v-calendar` 的 H5 适配版）
  - 月视图：有记录的日期显示小圆点
  - 点击日期跳转当天记录列表
- **可行性**：⭐⭐⭐⭐ 高
- **工作量**：3-4 天

#### 5. 草稿自动保存
- **背景**：当前表单页返回即丢失内容
- **方案**：
  - 输入时自动保存到 `localStorage['draft']`
  - 再次打开表单时提示恢复草稿
  - 提交成功后清除草稿
- **可行性**：⭐⭐⭐⭐⭐ 高
- **工作量**：1 天

#### 6. 批量操作
- **背景**：当前只能逐条删除/收藏
- **方案**：
  - 长按卡片进入批量模式
  - 多选后批量删除、批量移动分类、批量导出
- **可行性**：⭐⭐⭐⭐ 高
- **工作量**：2 天

#### 7. 回收站
- **背景**：删除后无法恢复，误操作风险高
- **方案**：
  - 删除后移至回收站（保留 30 天）
  - 回收站支持恢复/彻底删除
  - 30 天后自动清理
- **可行性**：⭐⭐⭐⭐⭐ 高
- **工作量**：2 天

---

### P2 - 差异化功能（竞争力提升）

#### 8. AI 辅助摘要
- **背景**：大模型 API 普及，可增加智能化能力
- **方案**：
  - 调用 OpenAI/Claude API（需用户自备 Key）
  - 功能：长文本自动摘要、情感分析、标签推荐
  - 纯客户端处理，不上传原始数据（隐私优先）
- **可行性**：⭐⭐⭐ 中（依赖用户 API Key 配置）
- **工作量**：3-4 天

#### 9. 时间线/故事模式
- **背景**：按时间轴展示记录，增强叙事感
- **方案**：
  - 时间线视图（类似微信朋友圈时间线）
  - 支持"那年今日"回顾功能
- **可行性**：⭐⭐⭐⭐ 高
- **工作量**：3 天

#### 10. 密码锁增强：生物识别
- **背景**：输入密码体验较差
- **方案**：
  - WebAuthn API（指纹/Face ID）
  - 降级方案：PIN 码（4-6 位数字）
- **可行性**：⭐⭐⭐ 中（WebAuthn 兼容性有限，iOS Safari 支持但 Android 浏览器参差不齐）
- **工作量**：3-4 天

#### 11. 模板系统
- **背景**：快速创建标准化记录
- **方案**：
  - 预设模板：购物清单、旅行计划、读书笔记、观影记录
  - 用户自定义模板
  - 模板一键套用
- **可行性**：⭐⭐⭐⭐⭐ 高
- **工作量**：2-3 天

#### 12. 数据可视化增强
- **背景**：当前仅有分类饼图
- **方案**：
  - 时间趋势图（每月记录数量）
  - 词云（从记录文本中提取高频词）
  - 热力图（类似 GitHub 贡献图）
- **可行性**：⭐⭐⭐⭐ 高
- **工作量**：3-4 天

---

### P3 - 探索性/远期功能

#### 13. PWA 离线支持
- **背景**：当前为普通 H5，断网不可用
- **方案**：
  - Service Worker 缓存核心资源
  - IndexedDB 离线读写
  - 网络恢复后自动同步
- **可行性**：⭐⭐⭐⭐ 高（PWA 技术成熟）
- **工作量**：4-5 天

#### 14. 多端同步（除 Gist 外）
- **方案**：
  - WebDAV 协议同步（兼容坚果云、Nextcloud）
  - 局域网直连同步（手机 ↔ 电脑）
- **可行性**：⭐⭐⭐ 中（WebDAV 实现较复杂）
- **工作量**：5-7 天

#### 15. 社交分享卡片
- **方案**：
  - 生成精美分享图（Canvas 绘制）
  - 支持分享到微信/微博（需后端支持）
- **可行性**：⭐⭐ 中（纯 H5 无法直接分享到微信）
- **工作量**：3 天

#### 16. Markdown 编辑模式
- **方案**：
  - 集成轻量 Markdown 编辑器（如 `simplemde`）
  - 预览模式实时渲染
- **可行性**：⭐⭐⭐⭐ 高
- **工作量**：3 天

## 四、技术债与架构建议

### 当前问题
1. **单文件过大**：`app.js` 已达 48KB，建议拆分为：
   - `core/app.js` - 应用入口
   - `modules/storage.js` - 数据层
   - `modules/ui.js` - 界面层
   - `modules/security.js` - 安全层
   - `modules/sync.js` - 同步层

2. **CSS 未模块化**：建议引入 CSS Variables 管理主题，减少重复样式

3. **无构建工具**：建议引入 Vite 进行打包压缩、代码分割

### 建议架构演进
```
universal_journal_h5/
├── src/
│   ├── core/          # 核心应用逻辑
│   ├── modules/       # 功能模块
│   ├── components/    # UI 组件
│   ├── utils/         # 工具函数
│   └── styles/        # 样式文件
── public/            # 静态资源
├── index.html
└── vite.config.js     # 构建配置
```
