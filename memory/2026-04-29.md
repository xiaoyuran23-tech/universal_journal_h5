# 2026-04-29 项目经验与教训

## 📦 项目：万物手札 H5 (Universal Journal)
**版本周期：** v3.1.0 → v3.4.0/v4.0.0  
**核心问题：** 同步后数据不显示、频繁语法崩溃、架构割裂

---

## 🔴 核心教训（血泪总结）

### 1. 自动化重构的陷阱
- **现象：** `edit_file` 对大文件局部修改时，因截断/缩进差异导致 `old_text` 匹配失败，反而破坏语法。
- **根因：** 自动化脚本无法完美处理嵌套模板字面量、嵌套大括号、异步回调等复杂 JS 结构。
- **教训：** 
  - ❌ 放弃 `edit_file` 进行复杂重构
  - ✅ 大文件采用 `write_file` 全量覆盖
  - ✅ 逻辑注入使用 Python 脚本 + Git 备份双保险
  - ✅ 每次修改后必须 `node -c` 校验语法

### 2. 架构演进不同步
- **现象：** 砍掉"分类"功能后，遗漏 `matchCategory` 变量引用、UI 残留 `item.category` 渲染、同步模块未更新数据流。
- **根因：** 多文件联动修改时缺乏全局影响面评估。改了 A，忘了 B、C、D 也依赖 A。
- **教训：**
  - ✅ 架构变更前，必须做**依赖图分析**（grep 搜索所有引用点）
  - ✅ 建立"架构契约"：数据结构变更后，渲染层、同步层、迁移层必须同步更新
  - ✅ 使用九子圆桌模式，强制多角度审查

### 3. 同步模块黑盒化
- **现象：** `CloudSyncV2` 被重命名但调用方未更新，导致 `TypeError: CloudSync.upload is not a function`。
- **根因：** 模块重构缺乏接口文档和类型检查，仅靠字符串替换不可靠。
- **教训：**
  - ✅ 公共模块必须明确定义 `window.XXX` 导出
  - ✅ 调用方必须检查模块是否存在后再调用
  - ✅ 同步流程必须有完整的错误捕获和重试机制

### 4. 安全漏洞被忽视
- **现象：** 加密功能被剥离，云端数据仅用 Base64 编码；`stripHtml()` 使用 `innerHTML` 可能触发 XSS。
- **根因：** 功能重构时，安全特性被当作"次要功能"丢弃。
- **教训：**
  - ✅ 安全特性（加密、XSS 防护）必须作为**架构基线**，不可剥离
  - ✅ 使用 `DOMParser` 替代 `innerHTML` 进行 HTML 清洗
  - ✅ 敏感数据传输必须端到端加密

### 5. 数据迁移不完整
- **现象：** 仅在启动时运行一次 `migrateLegacyData()`，云端同步回来的旧数据跳过迁移。
- **根因：** 迁移逻辑未覆盖所有数据入口（本地读取、云端下载、导入文件）。
- **教训：**
  - ✅ 数据迁移必须是**幂等**的，在任何数据入口都应触发
  - ✅ 建立数据版本标记，缺失版本标记自动触发迁移

### 6. 测试缺失
- **现象：** 每次推送都依赖用户手动验证，Bug 在上线后才被发现。
- **根因：** 缺乏自动化测试和集成验证流程。
- **教训：**
  - ✅ 本地必须走通核心流程：新建→保存→渲染→同步→下载→迁移→显示
  - ✅ 使用 `browser_use` 进行自动化 UI 测试
  - ✅ 建立"交付前检查清单"

---

## ✅ 重构原则（v4.0.0 必须遵守）

1. **单一数据源：** 所有数据操作通过 `StorageBackend`，禁止直接读写 `localStorage`
2. **接口契约：** 公共模块必须定义清晰的 `init()`、`export` 接口
3. **安全基线：** 加密、XSS 防护、数据迁移作为架构基线，不可剥离
4. **幂等迁移：** 数据迁移在任何入口都应自动触发
5. **全量覆盖：** 大文件修改使用 `write_file`，避免 `edit_file` 局部修补
6. **静态校验：** 每次修改后必须 `node -c` 验证语法
7. **九子审查：** 重大变更必须经过多角度审计
